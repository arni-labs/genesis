01 · One screen

What Genesis is

Genesis is a version-control server built on the Temper kernel. It speaks git's smart-HTTP wire protocol byte-for-byte and a structural subset of GitHub's REST v3, so git and gh work against it unmodified — but there are no bare repos on disk. Every git object — blob, tree, commit, ref, tag — is an IOA entity with its own state machine, and pull requests, reviews, and comments live in the same model as the objects they point at.

It exists for a specific workload. VISION.md calls it the Dark Factory:

"software systems where autonomous agents produce code continuously, at machine rate, with humans supervising through an evolution record rather than a ticket queue." — VISION.md

Three mismatches between that workload and the standard git + GitHub deployment motivated the design — VISION is careful to call them mismatches, not defects:

  • The primary writer is a program, not a human. Agents here run sandboxed, without network sockets or a forkable git binary, reaching out through capability-scoped host functions. Implementing smart-HTTP inside the sandbox is large; handing the sandbox a git binary undoes the isolation; neither integrates with the system's authorization and audit machinery.
  • Repository state lives in two places. Objects on a filesystem, PRs and reviews in a database behind a separate API. Every feature touching both must reconcile them. Here they share one substrate.
  • The audit record is partial. Git's reflog is local; GitHub's activity feed wasn't designed as a primary operational query surface. For a Dark Factory operator, "what did which agent do on which repository in the last hour" is the common case — it wants to run against the authoritative log.

Since the 2026-05-18 scope expansion, Genesis is also the app registry: apps are content-addressed (owner/app@hash), forkable with recorded lineage, installable into tenants, and servable in a multi-tenant commons mode. Genesis is the canonical home for Temper-native app bundles — production TemperPaw and Katagami installs come from it at pinned refs.

VISION keeps its own honesty: whether this beats a well-configured GitHub organization is an open question. What is not in question is that the wire is byte-exact — if a third-party tool works against github.com, it should work against Genesis, and if it doesn't, that's a bug.

02 · Containers

The substrate

One temper serve process. HTTP arrives at the kernel's router, which matches the path against registered HttpEndpoint entities and dispatches the bound WASM module. The handlers terminate the foreign wire — git pkt-lines, pack streams, REST JSON — and everything they persist goes back through OData as the resolved caller, so Cedar evaluates the real principal, not a system bridge.

Plate 02 — container view (C4 level 2)
Reading the plate

The load-bearing idea is the dispatch shape. HttpEndpoint is a first-class kernel entity (RFC-0001's K-1 kernel delta): registering one extends the router's match set, so terminating a foreign wire protocol is configuration, not a kernel patch. The handlers in the middle band are the only code that knows pkt-lines, pack format, or GitHub JSON shapes; the kernel below them knows only entities, events, policies, and OData. The arrow between the two carries the second load-bearing idea: handlers resolve the caller's GitToken themselves and pass a bridge_principal down, so every entity action a push or merge triggers is Cedar-evaluated as the actual agent — the audit row names who did it, with which scopes.

Component level — the push path
  1. git_receive_pack resolves the principal first: SHA-256 the bearer token, look up the GitToken row over /tdata/GitTokens. Anonymous push gets the standard git smart-HTTP 401 challenge. Successful resolution fires GitToken.MarkUsed — best-effort, so usage telemetry never blocks a push (ADR-0025).
  2. The incoming pack is stream-parsed — including OFS/REF delta entries and thin packs on the ingest side — each object decompressed, SHA-1'd, and shaped into a Blob/Tree/Commit/Tag row.
  3. scm_ingest_pack persists the rows as composite sub-writes and classifies each ref command: a bounded ancestry walk from the new tip toward the advertised old tip decides Ref.Update versus Ref.ForceUpdate (§05).
  4. Per-ref status goes back on the wire: ok refname or ng refname <reason>.
Component level — the clone path
  1. git_refs_advertise emits the pkt-line advertisement; byte parity against git-http-backend output is tested, not asserted (wire/tests/git_parity.rs).
  2. git_upload_pack parses want/have (NAK-only v0 — multi_ack and shallow negotiation are recorded follow-ups), walks the reachable object set with paged prefetch, and streams a pack-v2 of full objects with side-band-64k framing.
  3. Object bodies come from the raw-object cache first (§04); if a historically inconsistent repo advertises a ref whose target is not a stored commit, upload-pack answers with a git protocol ERR pkt-line instead of hanging the client (ADR-0011).
Why a kernel action bridge instead of an HTTP framework

Genesis's hard rules forbid host-side Rust handlers: every protocol surface is a WASM integration, dispatched through Temper's generic machinery. The payoff is that the git server inherits the kernel's whole governance story for free — Cedar on every transition, the event log as the audit surface, OData as the single query plane — and the kernel gains a generic reverse-proxy-into-entity-actions primitive any other Temper consumer can reuse. The cost is real too: byte-exact git serialization becomes Genesis's responsibility, verified continuously against the real git binary.

03 · The core bet

Objects as entities with state machines

A blob is a row, created by an action that carries canonical bytes and a content hash. For Blob, Tree, Commit, and Tag the entity Id is the SHA-1 — an OData GET by id is the natural git-object-by-hash lookup. Objects are immutable once hashed (Durable is their only state), with the integrity invariant Id == sha1(CanonicalBytes) checked on read.

The only mutable git surface is the ref — and ref updates are compare-and-swap transitions. At machine rate, two agents pushing the same branch must produce one accepted update and one cleanly rejected one, never a silent last-write-wins.

Plate 03 — compare-and-swap refs, and the event each transition leaves
Reading the plate

The CAS precondition lives in the entity's state machine, not in handler code: Ref.Update(PreviousCommitSha, NewCommitSha) rejects at the state level when the current SHA doesn't match. That puts concurrency control, policy (ForceUpdate needs the force scope), and audit on the same transition. The dark strip at the bottom is why the whole bet is worth it: the question "which commits did this agent produce in the last hour, and what did the policy engine decide about each" is a log query with the same filters and indexes as every other query in the system — not a filesystem walk joined to a separate audit store.

Tree entries are child rows with git's exact mode vocabulary (100644, 100755, 040000, 120000, 160000), sorted by git's canonical byte ordering — invariants declared on the entity, not conventions in code.

Why the event log is the audit surface

Git already works this way locally — the reflog is an event log — but the server usually isn't authoritative about history. Here every state change on every entity is an event, and current state is a projection over them. That buys replay for debugging, queryable history beyond any audit window, and an exact answer to VISION's "what happened on this repo at 3am last Tuesday." The recorded cost: log infrastructure is work GitHub has already solved at scale, and Genesis takes it on deliberately.

04 · Where bytes live

Storage tiers and the five speed mechanisms

ADR-0027 settled the storage split: entity rows carry metadata plus a content-addressed reference; all git object content lives in the object store. The row remains the authority — it owns the hash; store and cache are projections addressed by it. One source of truth, three places bytes can be read from.

Plate 04 — storage tiers and the clone read order
Reading the plate

Before ADR-0027, content at or under 128 KiB sat base64-inline on the row — which meant the authoritative event log was hauling file bytes it never needed for replay semantics, and row reads paid for content when only metadata was wanted. Dropping the inline threshold for object content to zero keeps the log lean. The migration is the same non-destructive pattern ADR-0011 established: new writes use the new shape, reads of pre-change rows resolve the legacy inline value and opportunistically backfill the store — nothing rewrites history, no migration job runs.

Five mechanisms make this fast. Each exists because a specific path was slow or fragile; none is speculative.

M1raw-object cache

Ready-to-emit object bodies at git-objects/{repository_id}/{sha}.b64, filled by pack ingest on write and backfilled by reads of older objects.

WHY — the clone path should not pay entity reads per object. Upload-pack reads the cache first and falls back to entity/field lookup only on a miss, so new repos are clone-ready immediately and old repos get faster without a destructive migration.

M2pinned-bundle install

GET /api/genesis/apps/{owner}/{name}/versions/{hash}/bundle returns the resolved app/dependency closure plus deterministic package files. Git fallback is off by default (an env flag re-enables it for admin recovery).

WHY — installs don't clone. Using clone as the install primitive made a large app pay git negotiation, object walking, pack emission, checkout, full reconcile, WASM persistence, and TemperFS bootstrap in one synchronous path that surfaced as a 120–300 s timeout.

M3field overflow staging

Field values over 128 KiB (FIELD_INLINE_MAX_BYTES = 131_072) are staged to the BlobStore; the row holds a content-addressed reference. ADR-0027 drops that threshold to zero for git object content; the 128 KiB line remains for non-object fields such as staged PackBytes.

WHY — WASM host-call buffers are finite. Values that ride host-call responses must stay bounded, or one large file breaks the dispatch machinery rather than degrading gracefully.

M4paged upload-pack prefetch

The pack-emission prefetch pages its selected object fields instead of asking OData for everything at once.

WHY — a large repo's worth of object fields in one OData payload can overflow the WASM host-call response buffer. Paging bounds every response, so repo size never becomes a protocol failure.

M5digest-aware reconcile

An app that is already installed, whose bundle digest and runtime state match, is adopted by recording Genesis provenance — specs, WASM, content, and seed bootstrap are not replayed.

WHY — warm restarts were re-installing everything; legacy rows without provenance repeatedly attempted first-time reconciles. Skipping unchanged pinned refs makes restart cost proportional to what actually changed.

Measured behavior on production

ADR-0011's verification record includes the Railway live clone smoke on a real production app: temperpaw/paw-patrol cold clone ~3 s, warm clone under 1 s, plus a read-only performance smoke (scripts/live-genesis-clone-performance-smoke.sh) measuring ls-remote, info/refs first byte, tiny clone, and large warm clone. Upload-pack emits phase timing — request parsing, reachable-object walking, pack emission — so a slow clone is attributable instead of opaque. The Genesis-vs-GitHub benchmark in docs/PERFORMANCE.md is the standing bar; §07 explains what happens if clone misses it.

05 · RFC-0004

The workflow layer

The PR, review, and merge entity model — with Cedar rules like "the author of a PR cannot approve it" — was merged long before anything could call it. RFC-0004 is the layer that makes those state machines reachable from real tooling, and it closes two enforcement holes first: push performed no authentication, and non-fast-forward updates were never classified, so force-push protection existed on paper only.

Auth lands first deliberately: every endpoint added afterward is born behind the token model instead of retrofitted. Push resolves the caller's GitToken (Basic username, Bearer, or gh-style token forms), challenges anonymous pushes with the standard 401, and passes the resolved principal through the action bridge so Cedar evaluates the real agent (temper ADR-0138). Force classification is an ancestry walk: if the advertised old tip is not an ancestor of the new tip, the sub-write dispatches Ref.ForceUpdate, which Cedar only permits with the force scope. Fast-forward pushes — the overwhelmingly common case — pay one ancestor check that terminates quickly on the first-parent chain.

On top of that sit the REST shims (/api/v3: repos, branches, refs, pulls, reviews, merge — each shipped endpoint shape-tested against a recorded github.com fixture) and the one genuinely hard piece: the server-side merge engine.

Plate 05 — scm_merge_pr: three strategies, one refusal
Reading the plate

The merge engine is the riskiest code in a system whose product is byte-exact compatibility — it is the one place the server authors new git objects. The design keeps the risk box small: the byte primitives (commit_canonical_bytes, tree_canonical_bytes in crates/git_object) were already hash-parity-tested including two-parent merge commits, the sub-write shapes were already proven by pack ingest, and the merge is accepted only when the three-way resolution is clean at tree level — no path modified on both sides since the base. Divergent content gets a 409 that says exactly what to do. Refusal beats guessing because the failure mode of refusal is a retry, while the failure mode of guessing is a wrong merge commit in a permanent, content-addressed history — and agents already have working conflict tooling locally (ADR-0024).

Shape compatibility is hard rule 7: every shipped endpoint carries a test comparing a recorded github.com fixture with the Genesis response — field names, types, required-field presence.

Cedar rules that switched from paper to live

With callers in place, the dormant policies became enforcement: the author of a PR cannot approve it (the CI gate exercises this with a second token), merge is reachable only from Approved, anonymous push is challenged, non-fast-forward without the force scope is denied, and deleting or force-updating the default branch keeps its deny policy. One intended behavior change rode along: existing automation that rewrites history must re-mint tokens with the force scope (ADR-0025 calls this out explicitly).

06 · RFC-0005

Directed Evolution, riding the substrate

Directed Evolution improves a running application the way a breeder improves an organism: under explicit, recorded selection pressure. It is not a separate system bolted onto Genesis — the control plane is itself a Genesis app (apps/directed-evolution), every concept a Temper entity with a state machine, installed by pinned refs and iterated by hot load. Genesis stores evolution state and app versions; it never runs agent work itself.

Plate 06 — the evolution loop, the execution bridge, and what watches it
Reading the plate

Follow the green line: a promoted winner becomes the new parent version, and the loop closes on the same organism. Three planes keep responsibilities clean. The control plane is entity state in Genesis — signals (scout sweeps of Datadog per organism runtime service, fingerprint-deduped so a repeating error makes one signal, not direction spam), pressures, directions, episodes, trials, eliminations, promotions, lineage. The execution plane is the bridge band: Genesis queues shared work items; a local worker claims and runs them; structured results route back through a receipt entity into a WASM router. The runtime plane is wherever the organism actually runs — its target (base URL, tenant, Datadog service, auth env-var names, never values) is data on the Organism row, so a second organism needs one governed configuration action, not a code change (ADR-0026). Mission Control observes all of it from entities alone; if live data is missing, it says so rather than synthesizing.

Episode contracts are frozen, and why that matters

The human gate is chat negotiation materialized into one entity action: SubmitEpisodeStartRequest carries the full contract — adaptation goal, viability constraints, metrics, evaluation stages, elimination rules, scoring rules, selection statement. Before trials queue, the SimulatedUserPlan and SelectionProtocol must be frozen (Draft → Active → Frozen); there is no silent fallback to synthetic personas. No variant may modify its own evaluators, stages, or rules. The point is that selection is only meaningful if the criteria can't drift while the contest runs — and if the human disagrees with a winner, the contract was wrong: stop, revise, run another episode. Manual winner overrides are explicitly rejected.

The proof-surface discipline
  • Simulated users observe; evaluators judge. Simulated-user work items target Trial entities and may only record journeys, observations, friction, and blockers. Stage pass/fail belongs to evaluator roles targeting StageResult. A variant cannot pass because a simulated user said "passed."
  • Telemetry is fail-closed. A stage that declares Datadog-measured evidence fails unless the evaluator records structured evidence: query, time window, result count, interpretation, and what a zero result means. Raw links are secondary.
  • Thresholds are mechanical. Declared hard limits in elimination rules are enforced by the result router after evaluator output is parsed — an agent may explain a judgment, but it cannot talk its way past a declared threshold.
  • Every elimination explains itself: a dead variant has a readable cause, linked evidence, and the rules that killed it.

Mission Control always shows the active lane, why it was chosen, what may proceed automatically, and what is blocked on the human. Automation is visible by design.

Observability correlation, mechanically

Local workers and the deployed server emit to the same Datadog backend, and the join is mechanical rather than prompt-text: the kernel's observation-metadata support (temper ADR-0041) accepts an X-Temper-Observe-Metadata header and stamps temper.observation.* attributes onto spans and events. Directed Evolution join fields ride that channel — tenant, episode, direction, generation, variant, stage, trial, persona/run, work item, app ref, runtime ref, role. Key results are always materialized into entities; the UI never depends on opaque Datadog links, and Datadog is evidence, never a decision-maker.

07 · Recorded, not silent

Deliberately deferred

PARITY.md keeps the honest ledger: every "works" row names the test or smoke that proves it, and what isn't built is a recorded decision with a trigger for revisiting — not an omission discovered later.

pack delta emission

Clones currently ship full objects — roughly 5× the wire size of a delta pack. Ingest already handles deltas; emission is the deferred half.

Trigger: the Genesis-vs-GitHub benchmark in docs/PERFORMANCE.md. If clone latency misses the at-least-match bar and wire size is the dominant lever, delta emission comes into scope. Conditional, not abandoned.

webhook delivery

The Webhook entity exists (specs/webhook.ioa.toml); the delivery side is RFC-0001 Phase 2. Until then, the event log itself is the observation surface — which for agent operators is the primary one anyway.

GraphQL v4

RFC-0001 Phase 3. REST covers the agent toolchain (gh speaks REST for the workflow surface Genesis ships); GraphQL serves humans and modern CLIs and waits until the REST surface is proven.

3-way content merge

A deliberate divergence, not a gap: GitHub would merge non-overlapping hunks in the same file; Genesis returns 409 and tells the caller to resolve locally. The engine's failure mode stays refusal, never a wrong merge commit. Full blob-level merge is ADR-0024's recorded follow-up.

Other recorded follow-ups
  • multi_ack / shallow negotiation — upload-pack is NAK-only v0.
  • Rebase merge method — returns 409 with an instructive error; ADR-0024 records it.
  • Fine-grained per-repo permissions and collaborator roles — scope strings only today; the owner model follows RFC-0003.
  • Commons rate-limit enforcement (RFC-0003 Phase 6), multi-GB blob chunking, Git LFS, publisher signing, multi-commons federation.
  • The always-on Evolution Agent (continuous watcher, cross-project aggregation) — a separate future effort with its own RFC.
